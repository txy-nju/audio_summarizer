
《多模态视频内容总结 AI 工作流架构设计书》：

### 1. 业务逻辑建模 (Workflow Modeling)

**核心目标：** 解决单模态总结的局限性（例如：只看文字不知道画面里的演示动作，只看画面不知道演讲者的具体观点）。通过图文结合，生成具备时间线逻辑、画面描述与核心观点高度一致的深度视频摘要。

**状态定义 (State)：** 在此场景中，State 需要同时承载多模态数据、中间分析结果以及最终的评审状态。

Python

```
from typing import TypedDict, List, Dict

class VideoSummaryState(TypedDict):
    # 输入层数据
    transcript: str                 # 视频语音识别文本 (ASR/Whisper 输出)
    keyframes: List[Dict]           # 关键帧列表，包含 base64 数据及时间戳：[{"time": "00:15", "image": "base64_str"}]
    user_prompt: str                # 用户具体的总结侧重点（如：侧重产品演示、侧重理论讲解）
    
    # 中间态数据
    text_insights: str              # 语音文本的提炼结果
    visual_insights: str            # 关键帧画面的动作/图表解析结果
    
    # 输出与循环控制
    draft_summary: str              # 当前生成的融合总结草稿
    critique: str                   # 审阅节点给出的修改意见（如：图文不符、遗漏关键画面）
    revision_count: int             # 重写次数
```

### 2. 节点与模块设计 (Nodes & Modules)

为了提高效率和逻辑的清晰度，我们将采用**并行处理（Parallel Processing）**模式来分别处理视觉和听觉信息，然后再进行融合。

- **分析层 (Analyzers / Workers - 并行执行)**
    
    - **`text_analyzer_node`:** 纯文本处理节点。负责读取 `transcript`，提取核心观点、章节主题和金句。这一步旨在过滤口语化的废话。
        
    - **`vision_analyzer_node`:** 视觉处理节点。调用多模态大模型（Vision LLM），传入 `keyframes` 和对应的时间戳，要求模型描述画面中的关键动作、PPT 文本或场景变化。
        
- **综合层 (Synthesizer / Drafter)**
    
    - **`fusion_drafter_node`:** 核心的“组装节点”。接收 `text_insights` 和 `visual_insights`，根据时间戳或逻辑相关性进行“图文对齐”。例如：在讲到某个概念（文本）时，画面展示了什么图表（视觉）。基于此生成结构化的综合 `draft_summary`。
        
- **反思层 (Reflector/Evaluator)**
    
    - **`consistency_checker_node`:** 幻觉与一致性审查员。核对草稿中的内容：是否有“文本里没提到且画面里也没有”的幻觉？是否遗漏了重要的 PPT 画面解析？如果不合格，输出具体的修改点至 `critique`。
        

### 3. 工作流拓扑图描述 (Topological Logic)

在这个拓扑结构中，文本和图片的初步分析是解耦且并行的，以大幅缩短运行时间（响应延迟）。

- **START** -> 分发至 `text_analyzer_node` 与 `vision_analyzer_node` (并行运行)。
    
- **`text_analyzer_node`** 与 **`vision_analyzer_node`** 执行完毕后，汇聚 (Join) 到 -> **`fusion_drafter_node`**。
    
- **`fusion_drafter_node`** -> **`consistency_checker_node`** (进行草稿质量评审)。
    
- **`consistency_checker_node`** -> **[Conditional Edge: `route_after_review`]**
    
    - **分支 A (Approve)**: 若无明显遗漏且逻辑自洽，流向 **END**。
        
    - **分支 B (Needs Revision & Loop < 2)**: 发现图文融合生硬或有遗漏，流回 **`fusion_drafter_node`**，强制模型结合 `critique` 重新生成草稿。
        
    - **分支 C (Max Loops Reached)**: 若循环达到阈值，强制终止并输出当前最佳版本至 **END**。
        

### 4. 关键技术选型 (Tech Stack)

- **核心框架:** LangGraph (利用其并发执行多个 Node 的能力处理图文双轨)。
    
- **多模态大模型 (Multimodal LLM):** * 强烈推荐 **Gemini 1.5 Pro**。它拥有超大上下文窗口，且原生支持极强的多模态理解能力。你可以一次性将所有的视频关键帧（Base64 或原生 File URI）和长文本 Token 喂给它，它在做 `fusion_drafter_node` 时的图文对齐效果远超单模态模型拼接。
    
    - 备选：GPT-4o 或 Claude 3.5 Sonnet（处理图像表现也非常出色）。
        
- **文本切片与检索 (可选):** 如果视频长达数小时，文本极长，可在 `text_analyzer_node` 中引入 **RAG 机制**，使用 Text Embedding 将长文本分段向量化，分段生成摘要后再做 Map-Reduce。
    

这样设计的优势在于：**文本大纲提供逻辑主线，关键帧提供事实与细节补充**，反思层确保两者不会出现“图文打架”的情况，最终输出的总结报告质量将非常接近人类专家的水准。
