# 🚀 架构底层优化记录 (Optimization Log)

## 📌 [阶段里程碑]：提取层性能与成本优化 - 基于场景感知智能抽帧 (Scene Change Detection)

### 1. 优化背景与痛点
在最初的 MVP（最小可行性产品）设计中，视频提取层的关键帧抽取逻辑为**固定的基于时间间隔的抽帧 (Fixed Interval Sampling)**（例如：每隔 `DEFAULT_FRAME_INTERVAL` 秒抽取一张图片）。
这在实际的多模态应用中暴露出两大核心缺陷：
- **💰 极高的财务与性能成本 (Token Waste)**：在类似学术讲座的视频中，一张 PPT 可能会停留长达十分钟。固定抽帧会导致大模型（Vision LLM）接收到数百张几乎完全一模一样的图片，极大地浪费了输入窗口的 Token 并拉高了 API 延迟。
- **🚨 重要信息遗漏 (Information Loss)**：如果视频在两次固定抽帧的时间空隙（例如 1.5 秒处）内飞速闪过一行关键代码或一个重要的报错弹窗，固定频率抽帧将彻底错过这段对总结至关重要的视觉信息。

### 2. 解决方案：基于灰度直方图比对的智能抽帧算法
为了在解决上述痛点的同时不对系统性能造成影响，我重构了 `core/extraction/infrastructure/extractor.py` 中的 `extract_frames` 核心方法，引入了**基于 OpenCV 的轻量级场景感知算法**，具体策略如下：

- **动态场景突变检测 (Scene Change Detection)**：
  系统会在读取每一帧时，实时将其转化为灰度图，并计算归一化灰度直方图。通过 `cv2.compareHist` 比较当前帧与上一张已提取帧的色彩与结构相关性。当相关性 `< 0.90` 时，系统判定画面发生了“场景突变”（如 PPT 翻页、屏幕切换），从而**瞬时精准地抓取该帧**。
  
- **防抖保护机制 (Debounce / Min Interval Guard)**：
  为防止视频转场动画或主讲人快速动作引发 1 秒内抽出数十张毫无意义的过渡帧，我引入了 `interval`（最小防抖间隔，默认 2 秒）。即画面再怎么剧变，必须冷却两秒后才能再次抽帧。

- **防断层兜底机制 (Max Interval Fallback)**：
  当画面长达 10 分钟没有变化时（如讲座长篇大论），系统引入了 `max_interval`（默认 60 秒）兜底策略。只要距离上次抽帧长达一分钟，系统会强制抽取一帧，用作喂给大模型的“时间线心跳包 (Heartbeat)”，彻底防范大模型因为太久没收到画面而陷入时空断层的幻觉。

### 3. 重构带来的优势 (Value Delivery)
1. **📉 API 成本断崖式下降**：对于以长篇大论和固定画面为主的知识分享类视频，本算法成功剔除了 90% 的冗余静止关键帧，单次调用的 Vision Token 开销呈指数级下降。
2. **🎯 核心信息的零漏报**：对于那些“短平快”的代码操作流视频，只要画面发生了重大变化，系统会凭借场景感知识别精准“咬”住切换瞬间进行抽帧，彻底杜绝了关键信息的时空断层遗漏。
3. **⚡ 极低的本地算力开销**：灰度直方图计算是经典计算机视觉中极其轻量级 (O(1) 级别) 的操作，即便对于长达两小时的高清 60 帧视频，也不会对本地 CPU 带来任何可感知的解码延迟，完全符合《架构即法律》中“严禁过度工程”的指导思想。

## 📌 [阶段里程碑]：AI 工作流流控架构升级 - 多模态 Self-RAG 与企业级路由防线 (Multimodal Self-RAG & Robust Routing)

### 1. 优化背景与痛点
在单线流的图拓扑阶段，系统存在以下难以忽视的隐患：
- **无中生有的深层幻觉**：单次生成（One-shot Generation）极易让大模型在长上下文中迷失，从而产生脱离视频事实的幻觉捏造。
- **“魔法字符串”引发的脆弱性**：在 `router.py` 和 `graph.py` 的条件连线中，充斥着硬编码的路由指令（如 `"Approve"`, `"Needs Revision"`）。这在重构时极易引发牵一发而动全身的链式崩溃。
- **缺乏面对大模型“发疯”的兜底能力**：如果大模型输出未遵循格式，或网络异常引发 JSON 解析失败，脆弱的路由层会直接抛出 `AttributeError` 或陷入无限死循环，导致整条业务链路死锁宕机。

### 2. 解决方案：双重质量防线与防御性路由编排 (Dual-Grader & Defensive Routing)
在不引入庞大第三方封装库的前提下，我利用原生 Python 和 LangGraph 的特性进行了彻底的神经中枢级重构：

- **Multimodal Self-RAG (多模态自反思闭环)**：
  将单一粗糙的审查节点拆分为两个专职的守护节点：**幻觉评分器 (Hallucination Grader)**（拦截无中生有）与**有用性评分器 (Usefulness Grader)**（拦截偏题）。全面强制开启大模型的 `JSON Mode` 和 `temperature=0.0`，实现基于 SSCD（时空语义对抗解码）的机械级精准判决。
  
- **消灭魔法字符串 (Eliminate Magic Strings)**：
  在 `router.py` 顶部建立全局防腐常量库（如 `ROUTE_HAS_HALLUCINATION`）。所有节点跳转、图谱组装以及测试断言全部引用该常量，彻底锁死了拼写错误导致路由幽灵流转的可能。

- **极限异常兜底策略 (Fail-Open / Safe Fallback)**：
  在路由函数中强制引入强类型阻断校验：`if not score or not isinstance(score, str)`。面对上游不可控的大模型输出乱码或因网络断流产生的 `None` 值脏数据，路由层不再崩溃，而是坚定地执行“保守降级放行”策略，保障主总线绝对畅通。

- **严苛的上下文类型约束 (Type Hinting Integration)**：
  在测试用例与节点签名中全面跨模块引入 `VideoSummaryState` 强类型字典批注。

### 3. 重构带来的优势 (Value Delivery)
1. **🛡️ 生产级的高可用性 (Production-Ready Robustness)**：系统现在具备了企业网关级别的容错能力。面对恶劣的网络波动或 LLM 的 JSON 解析失败，系统会自动降级并完成流程流转，拒绝给用户抛出白屏或卡死状态。
2. **🧠 幻觉率与偏题率断崖式降低**：通过严格的 JSON 结构化审查与多轮次定点切除重写，最终生成的摘要无限逼近于完全基于客观源数据的专家级报告。
3. **🔧 极致的工程可维护性**：魔法字符串的消失与全面 Type Hinting 的覆盖，使得这段代码拥有了极高的可读性与防重构破损能力。未来的开发者（或面试官）阅读代码时，能瞬间体会到这是一个 Senior 级别、深谙工程化规约的健壮架构。

## 📌 [阶段里程碑]：前端交互体验 (UX) 跃升 - 全链路实时状态透传 (Real-time Status Streaming)

### 1. 优化背景与痛点
在早期的应用体验中，系统对用户而言处于极端的“黑盒状态”：
- **黑盒焦虑**：长达几分钟的视频物理提取、高密度 Whisper 语音转录以及多轮大模型博弈，全部被隐藏在一个死板的 `st.spinner`（加载转圈动画）中。用户极易误认为浏览器卡死或系统宕机。
- **排版不直观**：当最终的总结生成时，渲染位置处于折叠容器的底部，用户视线焦点容易被过程日志喧宾夺主。
- **高级机制被掩盖**：我们耗费巨大精力打造的 Self-RAG 多智能体审查纠错过程（如拦截幻觉打回重写），用户在前端完全无法感知，导致产品极客感大打折扣。

### 2. 解决方案：高阶闭包回调与 LangGraph 流式事件订阅
为了打破底层引擎与前端展示之间的厚重壁垒，我实施了全链路的防腐透传方案：

- **闭包回调级联注入 (Callback Injection)**：
  在前端 `app.py` 中利用 Python 闭包特性定义了更新 `st.status` 容器的局部函数，并将其作为 `status_callback` 参数，层层向下安全透传。它穿透了防腐层，直接深入到了最底层的 `VideoSource`（提取层）与工作流编排网之中。
  
- **LangGraph 原生流式执行 (Event Streaming)**：
  这是重构的灵魂。在 `api.py` 中，彻底废弃了阻塞且封闭的 `.invoke()` 调用，转而启用了 `workflow_app.stream(stream_mode="updates")`。这使得图状态机在运行时，每当一个 Agent 完成局部推演，都会实时 `yield` 抛出状态快照。

- **动态报警与映射路由**：
  在拿到流式更新块后，我在后端架设了监听网络。一旦监听到审查官节点给出 `score == "yes"` 或 `score == "no"`，立刻拦截并向前端抛出鲜红的 `🚨 [系统熔断]` 日志，将被迫重写的内幕展现在用户眼前。
  
- **前端排版空间解耦 (UI Decoupling)**：
  将最终报告的渲染逻辑 `st.markdown(summary)` 从进度容器的作用域中剥离到最外层。利用 `expanded=False` 在任务结束后自动收起日志瀑布流。

### 3. 重构带来的优势 (Value Delivery)
1. **👀 极致的透明度与信任感 (Observability)**：打破系统黑盒，让用户如同站在工厂玻璃窗外一般，实时看到每一张画面的提取和每一个智能体的抉择，大幅提升了企业级 AI 产品的体验上限。
2. **🧩 完美解耦的架构设计 (Decoupling)**：在这个方案中，底层核心业务代码依然保持纯净的无状态逻辑，仅仅只是暴露了一个 Hook 接口 (`status_callback`)。它完全不依赖 Streamlit 框架——未来即便业务后端迁移至 FastAPI 开发，或者前端改写为 Vue/React 配合 WebSocket 推送，底层核心代码连一个字母都不需要改动。
3. **✨ 视觉聚焦与降噪**：任务运行期间全屏展开状态面板展示极客流水线日志，任务圆满完成后又懂得“功成身退”自动折叠，将最高优先级的屏幕视觉空间永久让位给 Markdown 总结报告，完美契合了“内容为王、结果至上”的直觉设计理念。

## 📌 [阶段里程碑]：AI 认知能力跃升 - 多模态主动求知机制 (In-the-loop Multimodal Tool Calling)

### 1. 优化背景与痛点
在处理热门短视频或极具前沿性的技术讲座时，传统的大模型分析流暴露出了致命的**知识盲区 (Knowledge Cutoff)**：
- **热词与梗图盲区**：视频画面中经常出现近期爆火的网络热梗、二创表情包，或是音频中提及了刚发布的框架（如 "DeepSeek-V3", "mhc"）。由于大模型训练数据的滞后性，AI 通常无法理解其真实含义，导致强行脑补，生成与视频实际表达南辕北辙的荒谬结论。
- **被动的单向数据流**：此前的节点只是“死板地”接受提取层发来的文本和图片进行单向总结，遇到不懂的内容只能依赖系统设定的 `temperature=0` 被动丢弃或产生幻觉。

### 2. 解决方案：引入 ReAct 节点内循环与“以图生文”的智能检索降维打击
为了打破数据隔离，我们将 `text_analyzer_node` 和 `vision_analyzer_node` 升级为了具备 **ReAct (Reason + Act)** 能力的微观智能体 (Micro-Agents)：

- **统一的检索引擎 (`tavily_search`)**：
  为大模型绑定了基于 `Tavily API` (专为 LLM 优化的无广告搜索引擎) 的检索工具。当转录文本中出现生僻梗汇时，模型会主动中止分析，抛出工具调用指令。
  
- **[架构亮点] 视觉降维检索策略 (Image-to-Text Search Strategy)**：
  考虑到直接的逆向识图 API（如 Google Lens / SerpApi）往往存在需要公网 URL 限制、Base64 上传极其受限且额度高昂等工程痛点，我们设计了极具实用性的**“以图生文再查询”降维打击策略**。
  在视觉节点的 Prompt 中，我们明确教会 VLM（视觉大模型）：当遇到无法理解的画面时，**首先发挥其强大的原生视觉理解能力，将画面转换为巨细无遗的文本描述**（例如：“屏幕上是一只狗坐在着火的房间里喝咖啡，配文 This is fine”），然后将这串描述直接作为 Query 参数传给普通的 `tavily_search` 文本搜索引擎去寻找“梗百科”。这种设计不仅完美避开了昂贵的搜图 API 墙，更展现了极其优雅的工程解耦与降维思维。

- **节点内 ReAct 微路由循环 (Intra-node Loop)**：
  在两个分析节点内部，实现了一个 `while loop_count < max_tool_calls` 的循环架构。这允许模型“带着搜到的外挂知识重塑上下文”，直到它觉得疑惑全部解答完毕，才输出最终的洞察报告流向下一节点。

### 3. 重构带来的优势 (Value Delivery)
1. **🧠 从“被动接受”到“主动求知”**：大模型真正拥有了触手，面对未知的长尾数据不再盲目自信地捏造，而是主动查阅最新百科，极大提高了对流行文化、最新科技前沿视频的解析精准度。
2. **🛡️ 极致的工程收敛与降级策略**：通过复用轻量级的文本搜索来解决复杂的识图查梗痛点，砍掉了脆弱的第三方图床/搜图依赖。即便用户没有配置 API Key 或网络断流，系统也只会通过返回警告字符串让 LLM 放弃工具调用并退推至“利用内置知识盲猜”，保证了服务的生产级高可用。
3. **🔌 架构的高内聚低耦合**：将搜索动作与循环机制封闭在具体的 Worker 节点内部，完全没有破坏外围的图拓扑结构。这证明了我们的基础架构能够完美兼容单节点能力的内卷式升级。

## 📌 [阶段里程碑]：基于 Checkpoint 的时间旅行与会话恢复架构升级 (Time Travel & Session Persistence)

### 1. 优化背景与痛点
在传统的一次性视频总结链路中，系统虽然能够给出高质量报告，但在“会话连续性”与“细粒度回溯问答”两个维度上存在明显天花板：
- **一次性死文本问题 (One-shot Static Output)**：总结生成完成后，用户若想继续追问“14:20 这里画面中的架构图在表达什么”，系统并没有可靠的状态恢复能力，只能重新跑一遍完整链路，成本极高且浪费 Token。
- **上下文断裂问题 (Context Reset)**：没有 `thread_id` 级别的持久化后，前端刷新、用户切换页面或稍后回来看结果时，之前的分析状态全部丢失，无法形成真正的会话式产品体验。
- **缺乏精确证据回溯 (Lack of Evidence-grounded Retrieval)**：即便已有最终摘要，系统也无法基于某个精确时间点快速定位最近关键帧与该时间窗的 Whisper 语音证据，导致追问只能依赖模糊印象，容易再次诱发幻觉。
- **测试覆盖断层 (Validation Gap)**：如果没有对 checkpoint 恢复链路、同一 `thread_id` 的覆盖写入逻辑，以及服务层会话复用机制进行专项验证，Time Travel 功能极易在后续重构中悄然失效。

### 2. 解决方案：四阶段渐进式落地的会话持久化与时间旅行体系
围绕《多模态视频内容总结 AI 工作流架构设计书》中的 5.2 方向，我将该能力拆解成四个工程阶段，确保每一步都可验证、可回退、可扩展：

- **阶段 1：Checkpoint 基础设施接入 (Checkpoint Foundation)**：
  在 `core/workflow/checkpoint_factory.py` 中建立统一的 checkpointer 工厂，默认接入 `InMemorySaver`，并预留 `postgres` 后端分支作为生产环境扩展入口。同时在 `core/workflow/session.py` 中引入 `thread_id` 标准化与自动生成机制，确保每一次工作流执行都能绑定唯一会话标识。随后将 `graph.py`、`api.py` 与 `workflow_service.py` 串联起来，在 `workflow_app.stream(...)` 时显式传入 `configurable.thread_id`，使 LangGraph 的状态写入具备真正的会话语义。

- **阶段 2：Time Travel 检索与证据组装 (Evidence-grounded Time Travel)**：
  新增 `core/workflow/time_travel.py` 工具模块，实现三个关键原语：
  1. 时间戳解析（支持 `MM:SS` / `HH:MM:SS`）
  2. 最近邻关键帧匹配（Nearest Keyframe Matching）
  3. Whisper `verbose_json` 时间窗证据抽取（Transcript Window Extraction）
  在 `core/workflow/api.py` 中新增 `answer_question_at_timestamp(...)` 入口：它先通过 `thread_id` 从 checkpoint 读取 `channel_values`，再抽取目标时间窗的语音片段、对应时间点的关键帧，以及历史摘要片段，最后将这些“证据包”统一喂给大模型回答。若未配置 API Key 或 OpenAI 调用超时/异常，则自动降级为“证据直出模式”，保证功能不崩溃。

- **阶段 3：前端会话化挂载 (Session-aware Frontend UX)**：
  在 `app.py` 中引入 `st.session_state`，持久保存 `current_summary`、`active_thread_id`、`restored_thread_id` 与 `time_travel_answer`。侧边栏新增会话绑定区，允许用户粘贴历史 `thread_id` 并恢复为当前会话；总结完成后，系统会自动把本次生成的 `thread_id` 回填到前端，并在主界面新增 `Time Travel Q&A` 区域，支持用户输入时间戳、证据窗口和追问问题，直接调用 `VideoSummaryService.ask_at_timestamp(...)` 完成回溯追问。

- **阶段 4：测试闭环与恢复链路守护 (Testing Closure)**：
  新增多层测试体系，形成从工具层到集成层的完整防线：
  1. `tests/core/generation/test_time_travel.py`：验证时间戳解析、关键帧匹配、转录窗口抽取的边界行为。
  2. `tests/integration/test_time_travel_pipeline.py`：验证 `answer_question_at_timestamp(...)` 的正常路径、thread miss、无 API Key 降级、OpenAI 异常降级，以及提示词组装是否正确包含时间戳、证据和用户问题。
  3. `tests/integration/test_checkpoint_restore_flow.py`：验证首次总结后状态会被 checkpoint 持久化，且同一 `thread_id` 的二次写入会覆盖旧状态，Time Travel 始终命中最新版本。
  4. `tests/integration/test_workflow_service_session.py`：验证服务层 `ask_at_timestamp(...)` 会默认复用最近一次会话的 `last_thread_id`，保证前端无需重复手填。

### 3. 重构带来的优势 (Value Delivery)
1. **🧵 从“单次总结器”升级为“可持续对话的视频智能体”**：系统现在不再是一次性吐出 Markdown 后即遗忘上下文的流水线，而是具备了真正的会话恢复能力。用户可以围绕同一个视频反复发起多轮追问，且每次都能精确锚定到历史分析状态。
2. **🕒 真正的时间旅行式证据回溯**：通过 `thread_id + timestamp` 的组合，系统能够瞬间定位目标时间窗附近的关键帧和语音分段，让回答建立在明确证据之上，显著降低“凭印象脑补”的风险。
3. **🛡️ 高可用的降级策略与生产前瞻性**：即便没有 `OPENAI_API_KEY`，或者 OpenAI 调用发生超时、限流、网络异常，系统仍会退回到“证据抽取结果”而不是崩溃。与此同时，checkpointer 工厂已预留 `postgres` 扩展位，为未来跨进程、跨重启的持久化部署打好了防腐层。
4. **🧪 工程质量闭环真正补齐**：与只测试 Happy Path 的脆弱方案不同，本次改造不仅覆盖了工具函数的边界场景，还专门验证了会话恢复、最新状态覆盖、服务层默认复用会话等隐蔽风险点，使 Time Travel 架构具备了在后续演进中持续稳定的工程护城河。
5. **👀 前端体验从“黑盒摘要”升级为“可交互时空探索”**：用户现在能在页面中看到并复用 `thread_id`，能够在总结之后继续沿时间线追问具体片段。这种交互方式比单纯的总结展示更接近真正可用的 AI 产品形态，也更符合长视频分析场景下“先总览、再钻取”的认知习惯。

## 📌 [阶段里程碑]：Map-Reduce 多智能体编排落地（Send API + 双分支 Worker + 汇聚融合）

### 1. 优化背景与痛点
围绕 5.3 多智能体协同架构目标，在工程落地前系统面临以下关键问题：
- **长视频单节点认知拥塞**：音频与视觉都由大节点串行吞吐，时长上升后易出现“上下文注意力涣散”，总结稳定性下降。
- **并发模式不可控**：缺少可切换并发策略，难以做 A/B 对比，也无法在回退与试点之间快速切换。
- **分片级可观测与隔离不足**：原有路径以节点内并行为主，难以在图级追踪单分片处理状态，单分片失败定位成本高。
- **关键帧状态体积风险**：若直接推进 Send API，长视频下状态复制会放大内存与 checkpoint 压力。

### 2. 解决方案：双轨并发架构 + Send API 试点闭环
本阶段采用“可回滚、可灰度、可观测”的工程落地策略，完成以下核心改造：

- **并发模式开关化（可灰度）**：
  在配置层引入 `CONCURRENCY_MODE=threadpool|send_api`，默认保持 `threadpool` 稳定路径，支持按环境变量或调用参数切换。

- **Map-Dispatch 图级 fan-out（Send API）**：
  在 `map_dispatch_node` 后新增 `route_audio_send_tasks` 与 `route_vision_send_tasks`，按 chunk 生成 `List[Send]`，将音频与视觉分支均升级为图级任务分发。

- **双 Worker 节点落地（Audio/Vision）**：
  新增 `chunk_audio_worker_node` 与 `chunk_vision_worker_node`，每个 worker 仅处理单个 `current_chunk`，输出标准化 `chunk_results` 供 reducer 汇聚。

- **Reducer 汇聚策略稳态化（Fan-in）**：
  通过 `VideoSummaryState.chunk_results` 的 reducer 合并分支结果，确保按 `chunk_id` 聚合、`latency_ms` 字段深度合并、结果顺序可重建。

- **关键帧引用化兼容（降低状态膨胀风险）**：
  引入 `frame_file + keyframes_base_path` 双格式解析机制，支持从文件引用回补 base64，减少未来 Send API 全量迁移时的状态复制压力。

- **前端控制面补齐（产品可操作）**：
  在侧边栏新增并发模式选择（ThreadPool / Send API），并将 `concurrency_mode` 透传至服务层和工作流入口，形成“可见、可控、可验证”的用户面板。

### 3. 已解决的问题（Problem Solved）
1. **解决了并发策略不可切换问题**：现在可在不改代码的前提下切换 `threadpool` 与 `send_api`，支持低风险试点与快速回滚。
2. **解决了图级分片编排缺失问题**：音频与视觉均可按 chunk 图级 fan-out，避免单节点吞吐瓶颈。
3. **解决了分片上下文不完整问题**：每个 Send 任务携带 `current_chunk` 与 `current_chunk_base_item`，确保 worker 无状态执行仍能正确融合。
4. **解决了关键帧格式迁移风险**：统一 image/frame_file 双通道读取，兼容历史数据与新引用化数据。
5. **解决了前端不可控问题**：用户可直接选择并行模式，产品层具备实验开关能力。

### 4. 产品优势与业务价值（Product Advantages）
1. **🧠 长视频处理稳定性提升**：Map-Reduce 编排将“全局重压”拆解为分片并行任务，降低单次推理认知过载概率。
2. **⚙️ 面向生产的灰度发布能力**：双轨并发架构支持先试点后扩展，出现风险可一键回退到 threadpool。
3. **🔍 可观测性与诊断效率提升**：图级 worker 与指标日志让分片处理路径更透明，便于定位性能瓶颈与失败点。
4. **🧱 架构可演进性增强**：关键帧引用化与 reducer 机制为后续“单 chunk 级恢复/重试”与更细粒度编排打下基础。
5. **🧩 前后端一致的可操作体验**：并行模式从“后端隐藏参数”升级为“前端可控能力”，更适配真实业务实验与对比评估。

### 5. 验证结果与当前结论
- **功能完整性**：Send API 路径已覆盖音频分支、视觉分支与汇聚节点，链路可完整执行。
- **回归稳定性**：多轮全量回归中，`threadpool` 与 `send_api` 通过率一致（均为 119 passed / 2 failed / 1 skipped）。
- **失败归因**：失败主要来自外部 API 配额（Whisper 403）而非编排架构缺陷。
- **性能现状**：当前环境下 `send_api` 较 `threadpool` 有一定耗时回退（约 11%），建议保持 threadpool 为默认，send_api 作为可选试点模式继续优化。

### 6. 里程碑结论
5.3 多智能体编排已从“架构设计”进入“可运行、可测试、可灰度”的工程完成态：
- 已具备完整的音视频分块解析与汇聚能力；
- 已具备产品层并行模式控制能力；
- 已具备 A/B 基线对比能力与后续性能优化抓手。

## 📌 [阶段里程碑]：Send API 分片实时进度可视化（Chunk Progress Streaming）

### 1. 优化背景与痛点
在前一阶段完成 Send API 音视频分片编排后，系统虽然具备并行处理能力，但在产品交互层仍存在明显短板：
- **缺少分片级可视化反馈**：前端只能看到节点级日志，无法直观看到“已完成多少分片”。
- **并行收益感知弱**：用户看不到音频与视觉分支的并行推进过程，容易误判系统“卡住”。
- **问题定位成本高**：当分片处理出现延迟或不均衡时，前端无法快速识别是音频慢还是视觉慢。

### 2. 解决方案与实现
本次改造在不破坏现有 `status_callback` 文本日志兼容性的前提下，补齐了“后端结构化进度事件 + 前端进度组件渲染”双端链路：

- **后端：结构化进度事件发射（core/workflow/api.py）**
  - 在 `summarize_video()` 的 `stream_mode="updates"` 循环中，监听 `chunk_audio_worker_node` 与 `chunk_vision_worker_node` 更新。
  - 基于 `chunk_id` 做去重统计，分别维护音频完成集合与视觉完成集合，避免重试导致重复计数。
  - 实时计算 `audio_done / total_chunks`、`vision_done / total_chunks`、`overall_done / (2 * total_chunks)`。
  - 通过 `status_callback` 发送 `[[PROGRESS]]{json}` 结构化事件（`type=chunk_progress`）。

- **前端：进度面板渲染（app.py）**
  - 在状态容器中新增三条进度条：音频分片、视觉分片、总体进度。
  - 在 `update_status_ui()` 中解析 `[[PROGRESS]]` 事件并更新 UI；普通字符串日志仍按原逻辑输出。
  - 保持 threadpool 模式兼容：仅在 send_api 事件出现时刷新分片进度，不影响既有流程。

### 3. 已解决的问题（Problem Solved）
1. **解决了“并行不可见”问题**：用户可以实时看到音频与视觉两条分支的完成进度。
2. **解决了“进度不确定”问题**：从纯文本日志升级为百分比进度，执行状态更确定。
3. **解决了“重试误计数”风险**：使用分片 ID 集合去重，确保进度不会虚高或反复抖动。
4. **解决了“兼容性风险”问题**：保留原有日志通道，旧状态播报与新进度播报可并存。

### 4. 产品优势与业务价值（Product Advantages）
1. **👀 可观测体验显著提升**：用户可直接看到并行任务推进情况，显著降低等待焦虑。
2. **⚡ 并行能力价值可感知**：双分支并行不再是“后端黑盒”，而是可视化的产品能力。
3. **🛠️ 诊断效率提升**：通过音频/视觉分开进度，能更快识别瓶颈分支并指导优化。
4. **🧩 低侵入、强扩展**：采用结构化事件前缀协议，后续可继续扩展失败分片、重试次数、分片耗时分布等指标。

### 5. 当前结论
本次改造完成后，Send API 的核心优势之一“分片处理实时监控并前端展示”已在项目中落地，形成了从图级 fan-out 执行到前端可视化反馈的闭环能力。

## 📌 [阶段里程碑]：人类审查机制落地（HITL Gate + 两阶段工作流）

### 1. 优化背景与痛点
在人类审查机制引入前，系统虽然已具备分片分析、证据聚合与自动质检闭环能力，但在“关键输出可控性”和“业务可审计性”上存在明显短板：
- **自动成文风险高**：聚合结果会直接进入成文与审查节点，缺少人工确认环节，难以满足对结果可控性要求较高的业务场景。
- **人类意图无法结构化注入**：用户即使有补充指导意见，也无法在“聚合完成后、成文前”作为显式输入进入后续生成与评分逻辑。
- **流程职责边界模糊**：第一阶段图中混入成文与质量审查，会导致“分析阶段”和“生成阶段”耦合，增加维护与排障复杂度。
- **重复计算成本偏高**：当用户只想调整聚合稿方向时，缺少从审批点继续执行的机制，容易触发不必要的前置链路重跑。

### 2. 针对目标（参考工作流架构拓展方向）
围绕《多模态视频内容总结 AI 工作流架构设计书》中“在聚合后引入人类审查断点、支持动态指令再生成”的拓展方向（5.4），本次改造聚焦以下目标：
- **目标 A：职责解耦**：将“分片分析/证据聚合”和“全篇成文/质量审查”拆分为两个阶段，避免单图过载与职责混杂。
- **目标 B：强制人类在环**：在 `chunk_aggregator_node` 之后设置必须经过的人类审查关口，确保关键业务输出可被人工校正。
- **目标 C：可恢复可追溯**：基于同一 `thread_id` 保持状态延续，第一阶段产物可在第二阶段无损恢复并继续执行。
- **目标 D：最小侵入演进**：在保持现有提取层与并发编排主体稳定的前提下，完成能力升级并维持回归稳定性。

### 3. 方案与落地成果
- **成果 1：图职责切分完成**
  - 第一阶段图：`build_video_summary_graph` 仅负责“规划 -> 分析 -> 聚合 -> 人类审查关口”，并在 `human_gate_node` 后结束。
  - 第二阶段图：`build_finalization_graph` 负责“成文 -> 幻觉审查 -> 有用性审查（闭环重写）”。

- **成果 2：状态契约扩展完成**
  - 新增并贯通 `human_edited_aggregated_insights`、`human_guidance`、`human_gate_status`、`human_gate_reason` 四个 HITL 字段。
  - 第二阶段成文节点与有用性审查节点已消费人类编辑内容与人类指导意见，实现“人类意图可执行化”。

- **成果 3：API/Service 两阶段入口完成**
  - API 层形成 `analyze_video`（第一阶段，返回待审批包）与 `finalize_summary`（第二阶段，产出最终总结）的清晰语义。
  - Service 层入口同步升级为 `analyze_url_video`、`analyze_uploaded_video` 与 `finalize_summary`，调用链语义与阶段职责一致。

- **成果 4：前端人类审查闭环完成**
  - 前端新增“待审批聚合稿展示 + 人工可编辑 + 审批指导输入 + 一键进入第二阶段”流程。

  - 核心集成链路（状态消息、checkpoint 恢复、两阶段 finalize 流）已通过回归。
  - 在后续“职责收敛”迭代中，第一阶段已移除与第二阶段无关的初始化字段，进一步保证边界清晰。

### 4. 业务价值与架构意义
1. **🧑‍⚖️ 可控性提升**：关键输出在成文前必须经过人工审查，显著降低“自动化一次成文”风险。
